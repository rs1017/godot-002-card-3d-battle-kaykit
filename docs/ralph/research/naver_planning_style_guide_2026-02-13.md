# 네이버 레퍼런스 기반 기획서 작성 지침서

작성일: 2026-02-13  
적용 범위: 본 저장소의 모든 에이전트 기획 산출물 (`docs/ralph/plans/*.md`)

## 1. 목적
`https://blog.naver.com/d994411/222171000179`의 작성 팁을 실무형 규칙으로 변환해, 문서 품질과 전달력을 일관되게 맞춘다.

## 2. 핵심 원칙
1. 주제는 좁게 잡는다.
- 큰 주제(예: 장르 전체 제안) 대신, 짧은 기간에 구현 가능한 단일 기능/컨텐츠로 한정한다.

2. 읽는 사람 기준으로 쓴다.
- 게임을 모르는 사람도 이해 가능해야 한다.
- 채용/검토 담당자가 빠르게 판단할 수 있도록 핵심을 앞에 둔다.

3. 근거를 먼저 제시한다.
- 왜 필요한지(문제-원인-기회)를 먼저 적고, 그 다음 해결안을 적는다.
- 플레이 경험/레퍼런스/트렌드 중 최소 1개 이상으로 근거화한다.

4. 문서 폼은 깔끔하게 유지한다.
- 긴 문단 대신 짧은 문장과 명확한 소제목을 사용한다.
- 텍스트가 길어지면 플로우/표/요약 박스로 먼저 설명한다.

5. 핵심 재미를 명시한다.
- “이 기획의 재미 포인트”를 1문장으로 분리해 명확히 적는다.

6. 신규/역기획 병행 전략을 허용한다.
- 신규 기획서만 고집하지 않는다.
- 기존 게임/컨텐츠를 역분석한 근거를 함께 쓰면 구현 가능성과 설득력이 올라간다.

## 3. 문서 구조 템플릿 (강제 순서)
1. 표지/메타
- 문서명, 날짜, 태스크명, 버전, 작성자(또는 담당 에이전트)

2. 목차
- 대제목 기준으로 빠르게 훑을 수 있게 작성

3. 개요 (Overview)
- 대상 게임/시스템 요약
- 제안 배경(왜 필요한가)
- 목표와 성공 기준

4. 본론 (Main Design)
- 기획안 정의: 무엇을 추가/변경하는가
- 사용 시점: 언제/어디서 동작하는가
- 진행 흐름: 단계별 플로우
- 상세 규칙: 입력, 상태, 예외, 보상/결과
- 기대 효과: 유저 경험/지표 변화
- 핵심 재미: 한 문장 명시

5. 구현 가능성 (Feasibility)
- 수정 파일/경로
- 데이터 구조(필요 시 CSV 링크)
- 협업 포인트(UI, 아트, 사운드, 밸런스)

6. 검증 계획 (QA)
- 재현 가능한 테스트 시나리오
- 합격 기준(PASS 조건)
- 리스크와 대응

7. 결론
- 릴리즈 우선순위
- 다음 액션

8. 참고/가정
- 레퍼런스 링크
- 가정/제약 조건

## 3-A. 확장 목차 표준 (피칭/사업 검토용)
외부 공유용 기획서(팀 모집, 투자/퍼블리셔 검토, BM 논의)는 아래 12개 목차를 기본으로 사용한다.
1. 게임 개요 (Introduction)
2. 게임 컨셉 (Game Concept)
3. 스토리 및 세계관 (Story & Worldview)
4. 게임 디자인 (Game Design)
5. 핵심 시스템 설계 (Core Systems Design)
6. 게임 아트 및 스타일 (Art & Visual Style)
7. UI/UX 설계 (UI/UX Design)
8. 플레이어 진행 및 성장 시스템 (Player Progression)
9. 게임 레벨 및 콘텐츠 기획 (Level & Content Design)
10. 사운드 디자인 (Sound Design)
11. 멀티플레이 및 소셜 기능 (Multiplayer & Social Features)
12. 기술 스펙 및 플랫폼 (Technical Specifications & Platform)

## 3-B. 읽기 쉬운 작성 규칙 (강제)
1. 한 문단 3~4문장 이내.
2. 각 섹션 첫 줄에 결론 1문장 먼저.
3. 기능/시스템 설명은 문장보다 표/플로우를 우선.
4. 전문 용어는 첫 등장 시 한 줄 설명.
5. 독자가 3분 안에 핵심(무엇/왜/어떻게)을 파악할 수 있어야 함.

## 3-C. 항목별 최소 포함 요소 (요약)
1. 게임 개요: 제목, 장르, 플랫폼, 타겟 유저, 한 줄 가치제안.
2. 게임 컨셉: 핵심 컨셉 1~2문장, 차별화 포인트 3개 이내, 벤치마크 비교.
3. 스토리/세계관: 배경 요약, 주요 캐릭터, 지역/세력 구조.
4. 게임 디자인: 플레이 루프, 조작 방식, 모드, 카메라/시점, 차별화 요소.
5. 핵심 시스템: 전투/강화/아이템/경제 규칙, 계산식, 예시 시나리오.
6. 아트/스타일: 비주얼 키워드, 레퍼런스 방향, 캐릭터/배경/UI 스타일.
7. UI/UX: 메인 플로우, HUD 구성, 피드백 규칙, 사용성 테스트 계획.
8. 진행/성장: 레벨 곡선, 스킬/특성, 장비 성장 구조.
9. 레벨/콘텐츠: 난이도 곡선, 퀘스트 구조, 보스/엘리트 설계.
10. 사운드: BGM 방향, SFX 규칙, 보이스/더빙 범위.
11. 멀티/소셜: 협동/경쟁 모드, 친구/길드/채팅, 랭킹 규칙.
12. 기술/플랫폼: 엔진 선택 이유, 기술 스택, 최소/권장 사양, 포팅 계획.

## 3-D. BM/사업성 섹션 (추가 권장)
BM을 논의하는 문서에는 아래를 포함한다.
1. BM 유형 후보 2~3개 비교 (예: Premium, F2P, Battle Pass, DLC).
2. 유저 경험 리스크 (Pay-to-Win, 피로도, 규제 위험).
3. 선택 근거 (장르 적합성, 타겟 유저 소비 패턴, 운영 난이도).
4. 출시 시점 BM과 추후 확장 BM을 분리 명시.

## 3-E. 손익분기점(BEP) 시뮬레이션 템플릿
가격/매출 논의가 있는 경우 아래 표를 필수로 포함한다.
- 가정값: 팀 규모, 기간, 월 인건비, 기타비용(%), 플랫폼 수수료, 환불률.
- 총비용: 인건비 + 외주/에셋 + 도구/라이선스 + 마케팅 + 운영비.
- 전환율 시나리오: 보수/기준/공격(예: 0.5%, 1%, 2%).
- 가격 시나리오: 기본가, 런칭 할인, 프리미엄 에디션, DLC.
- 결과표: 예상 판매량, 순매출, BEP 달성 여부.

권장 계산식:
- 총개발비 = 인건비 + 기타비용
- 손익분기 판매량 = 총개발비 / (판매가 * (1 - 플랫폼수수료) * (1 - 환불률))

주의:
- 시뮬레이션 값은 '가정'임을 명시하고 확정 수치로 단정하지 않는다.
- 국가/플랫폼 규제(확률형 아이템, 환불 정책) 리스크를 별도 항목으로 분리한다.

## 3-I. 일정/예산/BM 작성 최소 기준
외부 공유 문서(모집/퍼블리싱/투자 검토)는 아래를 필수 포함한다.
1. 개발 일정: 마일스톤, 산출물, 담당, 완료 기준
2. 예산: 인건비/에셋/도구/마케팅/운영비 분리
3. BM: 기본 BM + 확장 BM(DLC/패스 등) + 리스크
4. 출시 계획: 플랫폼별 출시 순서, 할인/프로모션 계획

## 3-J. 시나리오/월드형 기획서 확장 규칙 (Reference Sample-2 반영)
스토리/맵 중심 프로젝트 문서는 아래 목차를 추가한다.
1. 시나리오 설정
- 장르, 주제, 기획 의도, 줄거리, 테마

2. 맵 소개
- 2D 평면도, 맵 연결관계, 범례/아이콘, 지형/해발 설명
- 맵 파라미터(구역명, 위험도, 자원량, 이동시간 등) 표

3. 지역 세부 소개
- 지역별 목적, 배경 서술, 게임적 특이사항, 자연환경, 참고 이미지

4. 오브젝트 세부 소개
- 인터랙션 오브젝트, 수집 오브젝트, 퀘스트 오브젝트 구분

5. 환경/랜드마크
- 지역별 백그라운드 규칙, 랜드마크 기능/의미, 플레이 영향

6. UI 상세
- 캐릭터 선택/생성, 탐색 HUD, 도감/기록 UI
- 각 화면의 입력 동선(입력 -> 반응 -> 결과) 1줄 요약

## 3-K. 문서 품질 보정 규칙 (OCR/스크랩 자료 대응)
샘플 레퍼런스가 OCR/스크랩 기반일 때 아래를 강제한다.
1. 깨진 문자/혼합 언어 노이즈는 정제 후 반영한다.
2. 손상된 표는 재작성하여 의미 단위로 복구한다.
3. 외부 링크 이미지는 로컬 `images/`에 저장 후 상대경로로 교체한다.
4. 의미 불명 문단은 `가정` 섹션으로 분리하고 근거 부족 표시를 남긴다.

## 4. 시각자료 규칙
- 외부 이미지 URL 직접 삽입 금지.
- 문서 옆 `images/` 하위에 저장 후 상대경로 링크 사용.
- 예: `![전투 흐름](images/battle_flow_01.png)`

## 5. 작성 품질 체크리스트
- [ ] 문서 시작 1페이지 이내에 문제 정의와 목표가 보인다.
- [ ] 개요만 읽어도 기획 의도가 이해된다.
- [ ] 본론에 단계형 플로우 또는 표가 1개 이상 있다.
- [ ] 핵심 재미 포인트가 한 문장으로 고정되어 있다.
- [ ] 구현 파일 경로가 명시되어 있다.
- [ ] QA PASS 기준이 정량 또는 명확한 조건으로 있다.
- [ ] 참고 링크/가정이 문서 끝에 정리되어 있다.

## 6. Ralph 연동 규칙
- Ralph 사이클에서 생성되는 기획서는 본 지침서의 구조를 기본값으로 따른다.
- 사이클 완료 판정 전, 최소한 `핵심 원칙`과 `작성 품질 체크리스트`를 검토한다.
